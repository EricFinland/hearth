## The one-liner

hearth stops being a box where you drive agents, and becomes a box that **drives itself, improves itself, and runs a workforce while you sleep** — safely, because every change it makes to itself is a NixOS generation it can roll back. You are mission control, not the operator.

## What makes this cool (the "whoa")

The jaw-drop is this: **you type one sentence, and the computer rewrites itself in front of you without being able to brick itself.**

> "Give yourself a Discord notifier and turn it on."

You watch the map. A manager agent decomposes the goal and spawns a swarm of specialists. They write a NixOS module + the code, build it as a **new generation**, boot a sandbox VM of hearth's proposed next self, health-check it, and only then promote the change to the live host — with a watchdog that boots the previous generation if anything fails. The box became a better version of itself, from one sentence, and the failure mode is "nothing happened," not "I lost my laptop."

Almost no other system can do autonomous self-modification safely. NixOS can, because generations are immutable and atomic. That single property is hearth's unfair advantage, and nobody is showcasing it this way.

## Why hearth specifically can build this

It already has the three hard pieces, shipped and verified:
- **Full-machine reach** — agents run as operator with sudo; they can already run `nixos-rebuild`.
- **An audit trail** — every tool call, decision, and mode change is recorded.
- **Permission modes + approvals + kill switch** — plan/auto/bypass, a pending-approvals queue, and `/stop-all`.

Plus the spawn infra (queue → systemd units), the cockpit + live map, and web tools. The self-operating vision is not a rewrite; it is the natural compounding of what exists.

## The four pillars (all chosen)

1. **Self-evolving host** — the flagship. The box reads its own flake, proposes a change, proves it in a sandbox, and promotes it with auto-rollback.
2. **Agent swarm / mission control** — a manager agent decomposes goals and spawns a coordinated team; the tycoon map becomes a live org chart of agents building together.
3. **Always-on autonomous daemon** — hearth wakes itself on triggers (schedule, webhook, email, git push, file change) and runs standing missions while you sleep, then reports.
4. **The appliance** — once the loop is proven, package it: one-command install, a marketplace of agent tools, a gorgeous first-run, so other people can flash "the local-AI box."

These layer in that order. The swarm powers the flagship; the daemon is what kicks off self-improvement missions unattended; the appliance is how the proven thing reaches the world.

## The spine: NixOS rollback as the safety net

The whole vision is only sane because of one mechanism. Every self-modification is:
1. **Built**, never blindly switched — `nixos-rebuild build` produces a new system closure without touching the running one.
2. **Sandbox-tested** — the closure is booted as a VM (or on a spare node) and health-checked before it ever touches the host.
3. **Promoted with a watchdog** — on the host, `nixos-rebuild test` activates without setting the boot default; a health timer confirms the box is still healthy and reachable; if not, a watchdog boots the last-good generation.
4. **Always reversible** — the previous generation is one boot away, forever.

## Premises (agreed in session)

1. NixOS generations are a reliable enough safety net that autonomous self-modification is contained, not reckless. **Agreed.**
2. The swarm can produce valid Nix + code via an iterate-on-the-build-error loop, with a stronger model allowed for planning if the local 7B struggles. **Agreed, with the build/test gate as the backstop.**
3. The agent already has the reach to rebuild the host (operator + sudo). **Agreed — done manually today.**

## The real danger and how we contain it

An autonomous agent rebuilding the **running** host can take the box down. On the **blade specifically**, a change that breaks WiFi loses SSH and bricks it until physical access. Containment:
- **Self-evolution happens in a sandbox** (VM or a spare/second node), **never on the flaky blade directly**, until trust is earned.
- **Protected modules** — networking, bootloader, and the SSH/admin path are off-limits to autonomous changes without explicit approval (they go through the approvals queue, never bypass).
- **Watchdog auto-rollback** on the host: health check within N minutes or boot the previous generation.
- **Bounded blast radius** — a self-improvement mission runs in `auto` mode by default, so risky steps surface in the approvals queue.

## Approaches considered

### Approach A: CI-loop self-evolution (Phase 1, ships first)
The swarm edits the flake on a branch and pushes; the existing GitHub Actions eval gate proves it evaluates/builds; the change lands (auto-merge on green or your one-click). No live auto-rebuild yet.
- Effort: M. Risk: Low. Reuses: spawn infra, the agent loop, the existing CI eval gate, git.
- Proves the hardest unknown ("can the swarm write valid Nix end to end?") with zero risk to any running box.

### Approach B: VM-sandbox self-evolution (Phase 2, the flagship)
hearth builds its proposed next self, boots it as a NixOS VM, runs an in-VM health check, and only then promotes to the host via `nixos-rebuild test` + watchdog auto-rollback.
- Effort: L. Risk: Med. Reuses: the `.#image`/VM build the project already produces, the audit DB for health signals.
- This is the demo nobody else can do. Built on top of A, so the Nix-writing loop is already trusted by the time the live host is in scope.

### Approach C: Live test + watchdog (rejected as the starting point)
Skip the sandbox; `nixos-rebuild test` on the host with a health timer + rollback watchdog.
- Effort: M. Risk: High. A bad change can disrupt the running box before rollback; unacceptable as the first step, and especially on the blade.

## Recommended approach

**Staged A → B.** Ship the CI-loop first (fast, safe, proves the swarm-writes-Nix loop), then build the VM-sandbox flagship on top, with C's watchdog as the host-promotion safety once B is solid. Self-evolution targets a VM/spare node, not the blade, until trust is high.

## Success criteria (the "whoa" demo, staged)

- **Phase 1 done:** "Add a small module to yourself" → a swarm writes it on a branch → CI goes green → it lands, with the whole run visible in the cockpit transcript.
- **Phase 2 done (flagship):** one sentence → swarm builds the change → hearth boots a VM of its next self → health check passes → host promotes via test + watchdog → the new capability is live, and a forced-failure change auto-rolls-back with the box unharmed.
- **Pillar 3 done:** a standing mission ("keep my repos green") runs unattended on a trigger and produces a morning report.
- **Pillar 4 done:** a second machine is flashed from a hearth image and reaches a working swarm through a first-boot wizard.

## Distribution plan (the appliance pillar)

Reuse the existing `image`/`image-minimal` qcow-efi builds and CI. Add: a first-boot wizard (set admin key, pull models, pick a starter mission), a versioned release on GitHub Releases, and an agent-tool "marketplace" that is really just a declarative `hearth.agents.tools` list other people can extend. Ship it as "flash this, get a private self-operating AI box."

## Open questions

- Which model plans the Nix changes — local qwen2.5-coder with the iterate loop, or an API model for planning only (credentials channel already supports keys)?
- Health-check definition: what makes a generation "good"? (services active, port reachable, a smoke mission passes?)
- Sandbox substrate: nested VM on the blade vs a dedicated second node / the Proxmox target.
- Swarm coordination protocol: how the manager and specialists share state (extend the DB transport, or a richer message bus?).

## Next steps (build order, when you move to implementation)

1. **Swarm primitive** — a manager-agent loop that decomposes a goal and spawns specialist runs (extends the spawn infra + the agent loop). The map renders the tree.
2. **CI-loop self-evolution (Approach A)** — the swarm targets the flake on a branch; CI is the gate; land on green.
3. **VM-sandbox + promote + watchdog (Approach B)** — the flagship.
4. **Trigger daemon** — schedule/webhook/git-push triggers that launch standing missions.
5. **Appliance packaging** — first-boot wizard + release + tool marketplace.

## The assignment

That single sentence becomes the Phase-1 acceptance test: when the swarm can take *that* sentence and land the change through CI, the loop is real. Don't build anything yet — just write the sentence and keep it.
